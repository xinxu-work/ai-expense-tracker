How the Chat Works
The chat panel uses Claude API + tool calling — same concept as the Databricks agents you built:
User: "Which category did I overspend on?"
  → Claude calls: GET /summary/monthly?year=2026&month=3
  → Claude replies: "Your dining out was $780 against a $500 budget (156% utilisation)"

User: "Show me the trend for groceries"
  → Claude calls: GET /transactions?category_id=<groceries-uuid>
  → Claude replies: "Groceries trending up: Feb $420, Mar $455, Apr $510"
  → Dashboard auto-filters to grocery category

Raw data enters Supabase via Python (reading Excel, mapping columns, inserting through REST API), then dbt runs SQL transformations directly inside PostgreSQL — staging views clean and standardize the data (AEST timezone, date extraction), mart tables compute analytics (monthly summaries, savings rate, budget comparisons) — all without moving data out of the database, and the final tables feed directly into Power BI dashboards.

In plain English:
FastAPI = the front door. You POST a new transaction, it writes to the transactions table. That's it — raw data in.

dbt = the kitchen. It reads those raw tables, runs SQL queries (like your stg_transactions.sql above — joining transactions to dim_date, adding fiscal periods, cleaning nulls), and writes the results into new analytics tables like fct_monthly_summary. All pure SQL. No API calls.

Supabase = the database where everything lives. Both FastAPI and dbt connect to it directly — FastAPI writes raw data, dbt reads raw and writes analytics tables. They share the database, not each other.
